# メジャー更新時は新機能を opportunistic に取り込む

## ルール

依存ライブラリ・言語・ツールチェーンのメジャー更新 (major version bump) 時は、最小差分の breaking change 対応だけに留めず、**新バージョンで利用可能になった新機能のうち、本リポのコードベースを実際に改善できるものは積極的に採用する**。

## なぜ

メジャー更新は「更新コストを一度払うタイミング」。同じコードベースを後でもう一度触るより、触っているついでに関連改善も取り込むほうが合計コストが低い。「最小差分原則」はバグ修正や通常の機能追加には有効だが、メジャー更新という文脈ではむしろ機会損失になる。

また、新機能を使わないまま「更新だけ」を繰り返すと、新バージョンの恩恵を受けられないままメンテナンスコストだけが増え続ける。

## 適用場面

### やる

- changelog の「New Features」セクションを全項目確認
- 各新機能について **「本リポで grep した結果、置き換え候補パターンが見つかるか」** を確認
- 見つかれば同じコミット / 同じ PR で採用
- **採用しなかった新機能についても plan / PR description に「見送り理由」を明記** (後で「なぜこれを使わなかったの？」と聞かれないように)

### やらない

- コードベースに該当パターンがないのに新機能を先回りで導入する (YAGNI)
- 新機能採用のために大規模リファクタを同じ PR に混ぜる (別 PR に切る)
- runtime polyfill が必要な新機能を、polyfill 無しで使う

## 実例

TypeScript 5.9 → 6.0 移行時:

- **採用**: `dom.iterable` の `dom` 統合 (全 tsconfig の `lib` から `dom.iterable` を除去)。Node 専用パッケージでは `lib: ["esnext"]` まで縮小
- **見送り**: `Temporal types` (date-fns を置き換える必要があり別 PR 級)、`#/` subpath imports (該当パターンなし)、`Map#getOrInsert` (grep 0 件)、`RegExp.escape` (`new RegExp(` 未使用)、`--stableTypeOrdering` (TS 7 への過渡フラグでスコープ違い)

見送り項目も plan に書いておくと、レビューで「本当に該当パターンがないか」をレビュアーに再確認してもらえる。

## PR レビューでの適用

- メジャー更新 PR のレビューでは「新機能の採用機会を検討したか」を確認
- plan / PR description に「採用した新機能」「見送った新機能と理由」が書かれているか確認
- 不足していれば指摘
