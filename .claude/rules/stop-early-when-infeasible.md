# 実行不能と判断したら早期停止してレポートする

## ルール

調査・計画段階で「このタスクは現状のコードベースでは実行不能 (あるいは膨大なコストがかかる)」と判断したら、**作業を止めて状況と判断根拠をレポートする**。サンクコストに引きずられて無理に進めない。

## なぜ

「とりあえず進めてから問題が出たら相談」は、実行不能なタスクにおいてはコスト増を招く。早期に「不能」と判断できれば:

- 無駄な変更を積まずに済む
- ロールバックが不要になる
- 代替案や前提の見直しに時間を使える
- レビュアー/依頼者が状況を早く把握できる

特に依存ライブラリのメジャー更新では、「該当バージョンがそもそも peer dependency と衝突する」「必須機能が削除されている」など、コードを書き始める前に判明する不能条件がある。

## 適用場面

### 早期停止すべきサイン

- peer dependency の解決ができない (例: vue@3 固定 + vue-compat が非対応)
- 削除された機能を本質的に使っていて、移行先が存在しない
- 移行先バージョンが alpha/beta で本番投入に耐えない
- ランタイム/Node バージョンが対応していない
- 必要なポリフィル/adapter が存在しない

### レポートに含めるべき内容

1. **何を試したか** (調査した changelog, 確認したファイル, 実行したコマンド)
2. **なぜ不能と判断したか** (具体的なエラー / 非互換の根拠)
3. **どうすれば可能になるか** (前提条件・必要な先行作業)
4. **スコープを狭める代替案**があれば併記

## やってはいけないこと

- 「なんとかなるかも」と workaround を積み重ねて PR を肥大化させる
- 途中で諦めてブランチを放置する (必ず stop した旨を明示的に伝える)
- 不能判断を遅らせて、サンクコストが気になり引き返せなくする
