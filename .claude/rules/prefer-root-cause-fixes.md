# 根本原因を修正する、workaround に逃げない

## ルール

問題に直面したとき、症状を抑える workaround ではなく、根本原因を取り除く対応を優先する。

## なぜ

workaround は短期的には楽だが、技術的負債を積み増す。「`rootDir` を明示追加して急場しのぎ」のような対症療法は、背後にある構造的な問題 (例: 不要に複雑な依存ツール) を温存してしまう。少し手間が増えても根本を潰したほうが中長期で楽になる。

## 適用場面

- ビルドツールやライブラリの挙動を逆手に取る設定の追加
- エラーを抑えるための `// @ts-expect-error` / `eslint-disable` の濫用
- CI で一部チェックをスキップする

これらを採用する前に、まず「この問題の根っこはどこか？」「根を抜けば設定追加自体が不要にならないか？」を自問する。

## 実例

本リポジトリでは TypeScript 6 の `rootDir` 既定値変更で `@vercel/ncc` のビルドが壊れた際、`tsconfig.json` に `rootDir: "./src"` を追加する workaround を提案していた。しかし最終的には **ncc → rollup への移行** という根本対応を選んだ。結果:

- `rootDir` 設定が不要になり、公式 `actions/typescript-action` テンプレートと整合する構成になった
- ncc の古い CJS ベースビルドから ESM 出力に切り替わり、バンドルサイズも縮小
- TS 6 の rootDir 問題自体が消滅

workaround で済ませていたら、将来 TS 7 が出たときに同種の問題が再発していた可能性が高い。

## 例外

- 本当に緊急でその場を凌ぐ必要がある場合 (本番障害対応など) は workaround で応急処置し、**必ず**後続 issue を立てて根本対応を約束する
- 根本対応のコストが圧倒的に高く、かつ症状が軽微な場合はトレードオフ判断

## PR レビューでの適用

- 設定ファイルに `skipLibCheck` / `ignoreDeprecations` / `// @ts-expect-error` 等が追加されていないか確認する
- 追加されている場合はコメントで「根本対応の選択肢を検討したか」を問う
